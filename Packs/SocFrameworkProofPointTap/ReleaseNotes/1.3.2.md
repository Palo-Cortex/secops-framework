## Correlation Rule â€” SOC Proofpoint TAP - Threat Detected (rule v1.4.0)

### What changed

**Consolidation.** Replaces the two 1.0.4 rules (one integration instance per rule) with a single rule covering both `messages delivered` and `clicks permitted` event types. Volume is controlled by a threat status gate (`active` / `malicious`) rather than rule splitting.

**Vendor-neutral alert_fields.** All alert_field keys use either standard XSIAM alert schema names or the `socfw*` correlation rule prefix. Zero vendor-branded key names in the core field set.

**`socfw*` email fields.** Twelve fields added for the V3 SOC Framework email chain (`Foundation_-_Normalize_Email_V3`): `socfwemaildeliveryaction`, `socfwemaildirection`, `socfwemailthreaturl`, `socfwemailthreattype`, `socfwemailthreatstatus`, `socfwemailthreatid`, `socfwemailclassification`, `socfwemailphishscore`, `socfwemailmalwarescore`, `socfwemailcampaignid`, `socfwemailclickip`, `socfwemailclicktime`.

**Backwards-compatible `proofpointtap*` shim.** Twenty-seven `proofpointtap*` alert_fields added alongside the `socfw*` fields to support `soc-phishing-investigation-1.0.5` playbooks and Proofpoint TAP layouts during the migration window. Twenty-two map directly from raw dataset fields. Five map from `bc_*` XQL alters (first-element extractions from `threatsInfoMap`):

- `bc_threatid` â€” `json_extract_scalar($[0].threatID)` â€” supports `proofpoint-get-forensics` (scalar `threatId` arg)
- `bc_classification` â€” `json_extract_scalar($[0].classification)`
- `bc_threaturl` â€” `json_extract_scalar($[0].threatUrl/threatURL)` â€” supports URL rasterization, DBot scoring, HTTP fetch tasks
- `bc_threatinfomap` â€” `json_extract($[0])` â€” returns parsed first-element object; enables `.threat` and `.threatType` sub-key access matching old `threatsInfoMap -> [0]` behavior; critical for the branch condition in Tasks 20â€“21 of `SOC_Proofpoint_Message_Delivered`

These fields are marked for removal when `soc-phishing-investigation-1.0.5` is decommissioned from all tenants.

### Suppression

| | Old rules (1.0.4) | New rule (1.4.0) |
|---|---|---|
| Field(s) | `fromAddress + headerFrom + toAddresses + sender + ccAddresses` | `GUID` |
| Duration | 1 hour | 24 hours |
| Scope | All events from same sender/recipients | Single delivery event only |
| Mass email effect | Suppresses all other recipients' alerts | None â€” each recipient has a unique GUID |

**GUID** is assigned per individual delivery event. Suppressing on GUID has zero effect on other recipients of the same bulk email. The prior sender-based suppression caused a mass marketing email to silence threat alerts for all other recipients from the same sender for the suppression window â€” this was the reason the old rule was disabled. The 24-hour window covers Proofpoint TAP's batch replay behavior where the same event can be re-delivered across multiple polling cycles.

### Modeling rules â€” intentionally excluded

The 1.0.4 pack contains two modeling rules on `proofpoint_tap_v2_generic_alert_raw`. XSIAM does not document behavior when two separately installed packs both register a User Defined modeling rule on the same dataset. Installing a second rule creates an undefined collision with installation-order-dependent XDM field results. Since `soc-proofpoint-tap-1.0.4` must remain installed during the migration window, its modeling rules provide XDM coverage. Modeling rules will be added to this pack at v1.5.0 once the old pack is decommissioned.

## Decommission Path

Remove the BC shim at v1.5.0 when all of the following are met:

1. `soc-phishing-investigation-1.0.5` removed from all tenants
2. V3 email chain (`Foundation_-_Normalize_Email_V3`, `SOC_Email_Analysis_V3`) confirmed active end-to-end
3. Old Proofpoint TAP layouts replaced with V3 layouts using `socfw*` fieldIds
4. `grep -r 'proofpointtap' Packs/*/Playbooks/*.yml` returns zero matches

Removal at v1.5.0: delete `bc_*` alter block from XQL, delete `proofpointtap*` section from `alert_fields`, add modeling rule `ProofpointTAPModelingRules_1_3`, uninstall `soc-proofpoint-tap-1.0.4`.

---

## Breaking Changes

None. All changes are additive. The two old 1.0.4 correlation rules remain installed and can coexist with the new rule while `is_enabled: false` is set on the new rule. Flip `is_enabled: true` on the new rule and disable the two old rules to cut over.
