## [1.3.1] - March 2026

### Changed

#### Single integration instance replaces two-instance split

The previous architecture required two separate `Proofpoint TAP v2` integration instances
(one for Clicks Permitted, one for Messages Delivered) due to rule rate-limiting behavior
in earlier XSIAM versions. This is no longer necessary.

A single instance with `Events to fetch: All` now handles both event types. The correlation
rule's `filter type in ("messages delivered", "clicks permitted")` gate provides the same
logical separation. Remove the old `Proofpoint TAP v2_Clicks_Permitted` and
`Proofpoint TAP v2_Messages_Delivered` instances after enabling the new instance.

#### Classifier and mapper removed

The `Proofpoint TAP Classifier` and incoming mapper have been removed. Field normalization
is now handled entirely within the correlation rule XQL via `alert_fields` mappings using
`socfw*` fields. The integration instance `Classifier` and `Mapper (incoming)` fields
should be left empty.

The `proofpointtap*` custom incident fields that the old classifier populated are replaced
by `socfw*` fields read natively by `Foundation_-_Normalize_Email_V3`. Remove
`IncidentFields/` and `Classifiers/` directories if upgrading from a local copy of the
1.0.x pack.

#### Correlation rule schema fixes

- Added `id:` and `ruleid:` fields (were absent â€” caused silent SDK validation gaps)
- Removed `rule_id: 0` (caused SDK to resolve `id` as null)
- `fromversion` corrected to `8.0.0` unquoted (correlation rule requirement)
- `is_enabled: false` â€” rule ships disabled for side-by-side validation; enable manually
  after disabling the system Proofpoint rule (Step 3 of POST_CONFIG_README)
- Restored `filehash: proofpointsha256` field mapping (present in 1.0.4, dropped in
  intermediate versions; required for WildFire lookup grouping)

### Fixed

#### Modeling rule installation failure

The modeling rule directory was missing the required `.yml` metadata file and the three
rule files were not in the required `ModelingRules/<RuleName>/` subdirectory structure.
The rule was not registering on the platform despite appearing to upload successfully.

Fixed: created `ModelingRules/ProofpointTAPModelingRules_1_3/` subdirectory containing
`ProofpointTAPModelingRules_1_3.yml`, `.xif`, and `_schema.json`. The `fromversion` in
the YML is set to `6.10.0` per the Palo Alto developer guide (not `8.3.1` or `8.0.0`,
both of which cause error 101704 on installation).

### Added

#### New XDM field mappings (modeling rule v1.5.0)

- `xdm.source.ipv4 = clickIP` â€” scalar IP of the clicking endpoint; feeds XSIAM network
  story stitching alongside `xdm.source.host.ipv4_addresses`
- `xdm.source.user.username = recipient` â€” identity of the user who received or clicked;
  feeds XSIAM identity analytics and authentication story stitching across sources
- `xdm.email.observation_timestamp = clickTime` â€” timestamp when the recipient clicked the
  malicious link; distinct from `delivery_timestamp`; null for messages delivered events

#### Schema additions (modeling rule)

Added 12 real TAP API fields that were missing from `schema.json`: `headerFrom`,
`headerReplyTo`, `replyToAddress`, `toAddresses`, `xmailer`, `phishScore`, `malwareScore`,
`spamScore`, `impostorScore`, `messageSize`, `threatTime`, `id`. These are declared for
future XDM mapping and to prevent SDK validation failures if referenced in the XIF.

### Pack

**Pack ID changed** from `soc-proofpoint-tap` to `socfw-proofpoint-tap`.

The previous pack remains installed under the old ID. Installing this pack does not
overwrite or remove content from the previous version. Both packs can coexist on the
same tenant, enabling side-by-side validation before cutover.

**Duplicate alert risk:** if old Proofpoint correlation rules and the new consolidated
rule are both enabled simultaneously, every TAP detection generates two alerts. Disable
all old rules before enabling the new rule â€” see POST_CONFIG_README Step 2.
