## [1.2.1] - March 2026

### Fixed

#### SOC Proofpoint TAP - Threat Detected — Double-encoding bug blocking all events (Critical)

The correlation rule was calling `to_json_string()` on `threatsInfoMap` and `messageParts`
before passing them to `json_extract_scalar()`. Both fields arrive from the TAP API already
serialized as JSON strings. Wrapping them in `to_json_string()` a second time produced
double-encoded strings (escaped quotes inside escaped quotes), causing all
`json_extract_scalar` calls to return `null`. The result was that `threatStatus`,
`threatID`, `classification`, `threatUrl`, and SHA256 from `messageParts` were empty on
every alert — the rule fired but carried no threat intelligence.

Fix: removed `to_json_string()` wrappers. Both fields are now used directly:
`threatsInfoMap` and `messageParts` are passed to `json_extract_scalar` and array
extraction functions as-is.

**Impact:** Every Proofpoint alert generated before this fix had empty threat fields.
`SOC_Email_Signal_Characterization_V3` and `SOC_Email_Verdict_Resolution_V3` downstream
would receive no `threatStatus`, no `classification`, and no SHA256, causing them to
route to low-confidence / benign verdicts regardless of actual threat content.

#### SOC Proofpoint TAP - Threat Detected — compress_window reduced from 2h to 30m

The `compress_window` was set to `2h`. Because TAP delivers events with timestamps up to
90 minutes in the past (batch delivery latency), a 2-hour compress window caused the XSIAM
HTTP Collector to treat valid events as stale and silently drop them. Reduced to `30m`,
which accommodates TAP batch latency while preventing replay of genuinely old data.

This was the root cause of Proofpoint events failing to appear in the tenant dataset during
PoV testing — the correlation rule and HTTP Collector were both correctly configured, but
events were being discarded before the rule could evaluate them.

#### SOC Proofpoint TAP - Threat Detected — Non-ASCII characters removed

Em-dashes and arrow characters in the rule description and comments replaced with ASCII
equivalents (`—` → `--`, `→` → `->`). Non-ASCII characters in correlation rule YAML cause
error 101704 on upload to XSIAM.

### Changed

#### SOC Proofpoint TAP - Threat Detected — Alert naming and suppression

**Alert naming** updated to `[Email]` prefix pattern consistent with the SOC Framework
cross-source grouping standard:

- Messages delivered: `[Email] {recipient} - Initial Access: Threat Email Delivered`
- Clicks permitted: `[Email] {recipient} - Initial Access: Malicious Link Clicked`

`to_string(recipient)` used to handle the TAP API returning `recipient` as an array.

**Suppression** on `GUID` with 24-hour window. `GUID` is the TAP campaign-level message
identifier — suppressing per GUID for 24 hours prevents duplicate alerts for the same
delivery event while preserving full blast-radius visibility across distinct GUIDs.
