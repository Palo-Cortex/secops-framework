## [3.6.0] - March 2026

### New

#### SOCActionClassMap_V3 — New list for multi-vendor action routing

`SOCActionClassMap_V3` is a new list in `soc-optimization-unified/Lists/` that maps every
SOC Framework action name to an action class (`endpoint`, `identity`, `email`, `indicator`,
`network`, `comms`). `SOCCommandWrapper` reads this list at execution time to determine which
vendor to route a command to when the alert source supports multiple integrations.

28 actions mapped across 6 classes. File placement:

```
Lists/SOCActionClassMap_V3/
├── SOCActionClassMap_V3.json
└── SOCActionClassMap_V3_data.json
```

### Changed

#### SOCProductCategoryMap_V3 — Multi-vendor `responses` map per source

Each dataset entry in `SOCProductCategoryMap_V3` now carries an optional `responses` object
alongside the existing `response` field. `responses` maps action classes to integration brand
strings, allowing a single alert source to route different action types to different
integrations — for example, a CrowdStrike Endpoint alert calling `CrowdStrike Falcon` for
`soc-isolate-endpoint` but `Active Directory Query v2` for `soc-disable-user`.

The existing `response` field is preserved unchanged for backward compatibility.

11 dataset entries updated with `responses` maps: `ds_crowdstrike_falcon_event`,
`ds_proofpoint_tap_v2_generic_alert_raw`, `ds_proofpoint_tap_v2_generic_alert`,
`ds_okta_systemlog`, `ds_paloaltonetworks_cloud_identity_engine_directory`,
`ds_microsoft_defender_identity`, `ds_panw_xdr_agent`, `ds_msft_defenderxdr`,
`ds_msft_graph_security_alerts`, `ds_trend_micro_vision_one_v3_generic_alert`,
`ds_paloaltonetworks_firewall_threat`.

#### SOCCommandWrapper — Multi-vendor action class routing

`SOCCommandWrapper` now uses a three-step vendor resolution:

1. Look up the action's class from `SOCActionClassMap_V3`.
2. Look up `SOCFramework.Product.responses.{class}` for the correct integration brand.
3. Fall back to `SOCFramework.Product.response` (legacy single-vendor, backward compatible).
4. Fall back to the first available vendor in the action's `responses` dict.

This change is backward compatible. Any deployment that has not populated `responses` in
`SOCProductCategoryMap_V3` will transparently fall through to step 3.

#### Foundation_-_Product_Classification_V3 — New task sets `SOCFramework.Product.responses`

Task 250 reads the `responses` map from `SOCProductCategoryMap_V3` for the matched product
key and writes it to `SOCFramework.Product.responses` in playbook context. This runs
immediately after the existing task that sets `SOCFramework.Product.response`, adding no
latency to the Foundation chain.

#### SOCFrameworkActions_V3 — CrowdStrike added to `soc-block-indicators`

`soc-block-indicators` now includes a `CrowdStrike Falcon` vendor response using
`cs-falcon-upload-custom-ioc` with `ioc_type: sha256`, `action: prevent`,
`platforms: windows`. In Shadow Mode this logs the IOC upload to the war room without
creating the custom IOC. In Execute Mode it creates a prevent policy for the hash in
the CrowdStrike console.

### Test Suite

`tools/run_tests.py` — New automated test runner. Sends replay scenarios via
`replay_scenario.py`, waits for playbooks to complete, then executes XQL assertions live
against the tenant via the XSIAM REST API (`start_xql_query` / `get_query_results`).
Reports `✓ PASS` / `✗ FAIL` per check with actual values. Credentials read from `.env`.

`scenarios/h1_h2_email_only.yml` — Proofpoint TAP only (H1, H2, N6).
`scenarios/h3_h4_h5_endpoint_only.yml` — CrowdStrike only (H3–H6, N1, N7).
`scenarios/h8_full_chain.yml` — Full Turla Carbon kill chain (H8, N2, N4).
`scenarios/n9_staleness_test.yml` — compress_window staleness confirmation (N9).

Usage:
```bash
python3 tools/run_tests.py --all              # send replays + assert
python3 tools/run_tests.py --all --skip-replay # assert against existing data
python3 tools/run_tests.py --test H4          # single test
```
