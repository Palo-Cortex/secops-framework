## SOC Framework Unified – Release Notes

### Version 3.7.5

#### Overview

This patch corrects three bugs in the Foundation layer. Two caused every alert
to take the Generic normalization path regardless of actual data source. The
third caused concurrent duplicate alerts to close each other due to a race
condition in the dedup logic, leaving cases frozen mid-playbook in the
Resolution Center. All three were silent — no visible error, failure only
apparent by tracing execution logs or observing stuck case state.

---

### Fixes

#### Foundation — continueonerror enforcement (11 playbooks, 20 tasks)

Every `regular` and `playbook` type task in every Foundation playbook must carry
`continueonerror: true`. This is a hard architectural requirement: Foundation
playbooks run on every alert and must never halt the chain due to a single task
error (e.g., an empty field, a missing list reference, or a context key that
hasn't been set yet).

20 tasks across 11 playbooks were missing this property. The most critical gaps:

- **Foundation - Upon Trigger V3** — 4 sub-playbook calls (Enrichment, Data
  Integrity, Dedup, Normalize Artifacts) were missing `continueonerror: true`.
  Any failure in these tasks would have silently stopped alert processing before
  normalization or analysis ran.
- **Foundation - Normalize Artifacts V3** — all 6 per-category normalizer
  dispatch calls (Email, Endpoint, Identity, Network, Cloud, Generic) were
  missing `continueonerror: true`.
- **Foundation - Error Handling** — the "Print Errors to War Room" task itself
  was missing the property (the error handler was itself stoppable on error).

Remaining affected playbooks: Data Integrity V3, Dedup, Environment Detection,
Get Alert Tasks and Store to Dataset V3, Product Classification V3, Case Sync,
Escalation, Performance Capture.

All 20 tasks now carry `continueonerror: true`.

---

#### Foundation - Product Classification V3 — CrowdStrike routing to Generic (fix)

**Symptom:** All alerts from `DS:CrowdStrike/Falcon_Event` were hitting the
Generic normalizer instead of the Endpoint normalizer. This caused downstream
NIST IR routing to fail to load Endpoint-specific context keys, MITRE tactic
routing, and CrowdStrike containment/eradication actions.

**Root cause:** The "Does List Product Category List Exist?" condition task
(task 24) evaluated `lists.SOCProductCategoryMap_V3` directly, but in XSIAM
this context path is only populated after an explicit `getList` call. Without
that call the condition always resolved empty, routing the flow to `#default#`
(Done) and skipping all classification logic. `SOCFramework.Product.category`
was never written, so Normalize Artifacts received an empty `ProductKey` and
fell through to the Generic branch on every alert.

**Fix:** Added task 32 (`getList name=SOCProductCategoryMap_V3`,
`continueonerror: true`) between the section title (task 21) and the condition
gate (task 24). Chain is now: `21 → 32 → 24`. The category map already
contained the correct entry (`ds_crowdstrike_falcon_event → Endpoint`) and the
tag transform was correct — the list simply needed to be in context for the
condition to evaluate.

### Tooling

#### tools/check_foundation_continueonerror.py (new)

CI guard that audits all Foundation playbooks in a pack and fails the build if
any `regular` or `playbook` type task is missing `continueonerror: true`. Exits
1 in `--ci` mode on any violation. Add to `soc-packs-pr-gate.yml`:

```yaml
- name: foundation-continueonerror
  run: python3 tools/check_foundation_continueonerror.py --pack Packs/soc-optimization-unified --ci
```

Supports `--pack`, `--file`, `--ci`, and `--all` (advisory scan of
non-Foundation playbooks).

#### Foundation - Dedup V3 — mutual close race condition (fix)

**Symptom:** When multiple duplicate alerts fired simultaneously, all concurrent
playbook instances would close each other. Cases appeared frozen mid-playbook
in the Resolution Center with no error — the investigation was closed underneath
a still-running playbook.

**Root cause:** Each instance called `closeInvestigation` targeting the *other*
similar alert IDs (`${DBotFindSimilarIncidents.similarIncident.id}`). With three
concurrent alerts A, B, C: A closed B and C, B closed A and C, C closed A and
B — all three closed, last writer won unpredictably. The "I'm the latest,
closing the others" framing was the wrong model entirely.

**Fix:** Replaced the close-others approach with deterministic self-close based
on minimum alert ID. Each instance independently computes `min(similarIDs +
myID)` → `SOCFramework.Dedup.WinnerID`. If `alert.id == WinnerID` the instance
is the canonical case and continues the chain. If not, it prints a self-close
note and calls `closeInvestigation(id=${incident.id})` — closing only itself.
All instances evaluate the same immutable data and reach the same conclusion.
No coordination, no lock, no race.

New task chain after similar alerts are found:
`16 → 22 (compute winner) → 23 (am I winner?) → 24/25 → 20 (DryRun gate) → 26 (close self) or Done`

Tasks 14 (close others) and 15 (old print) removed. Tasks 22–26 added.
DryRun gate preserved on the self-close path — smoke testing remains safe.
