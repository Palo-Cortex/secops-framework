## [3.5.0] - March 2026

### IncidentFields (new)

Added 12 generic email incident fields to `soc-optimization-unified`. These fields replace
vendor-specific `proofpointtap*` fields with a product-agnostic namespace, enabling any
email security source (TAP, Defender for O365, Mimecast, etc.) to write to a single
consistent contract consumed by `Foundation_-_Normalize_Email_V3` and the Email NIST IR
lifecycle.

All fields use the `socfw` prefix to prevent collision with Marketplace content. All fields
have `associatedToAll: true` making them available across all alert types.

| Field | cliName | Type | Purpose |
|---|---|---|---|
| Email Delivery Action | `socfwemaildeliveryaction` | Single select | Routing gate: delivered / blocked / click_permitted / click_blocked |
| Email Direction | `socfwemaildirection` | Single select | Containment scope: inbound / outbound / internal |
| Email Threat URL | `socfwemailthreaturl` | Short text | soc-block-indicator target, IOC grouping |
| Email Threat Type | `socfwemailthreattype` | Single select | Analysis branch: url / attachment / domain |
| Email Threat Status | `socfwemailthreatstatus` | Single select | Urgency gate: active / cleared / malicious |
| Email Threat ID | `socfwemailthreatid` | Short text | Vendor threat tracking reference |
| Email Classification | `socfwemailclassification` | Single select | phish / malware / spam / impostor |
| Email Phish Score | `socfwemailphishscore` | Number | 0‚Äì100, feeds Analysis.confidence |
| Email Malware Score | `socfwemailmalwarescore` | Number | 0‚Äì100, gates soc-detonate-file |
| Email Campaign ID | `socfwemailcampaignid` | Short text | Primary campaign grouping key |
| Email Click IP | `socfwemailclickip` | Short text | Lateral risk detection, click-source grouping |
| Email Click Time | `socfwemailclicktime` | Short text | Click event timestamp (ISO 8601) |

#### Breaking change
The `proofpointtap*` custom fields are removed from this pack. The TAP correlation rule
in `SocFrameworkProofPointTap` must be updated to write to `socfw*` fields before
deploying 3.5.0. See `feature/email-normalization-fields` PR for the coordinated rule update.

### Bug Fixes

#### SOCProductCategoryMap_V3 ‚Äî Missing `response` field on all entries (Critical)

The `response` field was absent from every entry in `SOCProductCategoryMap_V3`. This caused
`SOCCommandWrapper` to silently fall back to the first key in the vendor responses dict for
every alert source, executing the wrong vendor command regardless of which integration was
actually in use (e.g. `core-isolate-endpoint` firing instead of `cs-falcon-contain-host`
for CrowdStrike alerts in Full Run mode).

`response` is now populated for all 21 mapped sources. The `response` value is the
integration brand string that `SOCCommandWrapper` looks up in `SOCFrameworkActions_V3` ‚Äî
which may differ from the ingest dataset name (e.g. `ds_crowdstrike_falcon_event` ‚Üí
`"CrowdStrike Falcon"`). Sources with no response integration defined in
`SOCFrameworkActions_V3` are intentionally left without a `response` field.

#### SOCFrameworkActions_V3 ‚Äî Vendor key casing errors

Three vendor keys were mismatched against the strings `SOCFramework.Product.response`
actually produces, causing the same silent fallback described above:

- `CrowdstrikeFalcon` ‚Üí `CrowdStrike Falcon` (8 actions)
- `0365 Compliance` (zero) ‚Üí `O365 Compliance` (letter O)
- `Okta V2` ‚Üí `Okta v2`

#### value_tags.json ‚Äî Duplicate rows inflating dashboard metrics

Nine duplicate rows removed. `cs_falcon_run_script` in the eradication category had 4
identical rows, causing the Value Metrics dashboard to overcount CrowdStrike eradication
time savings by 4√ó. Additional duplicates removed: `disable_user`, `okta_clear_user_sessions`,
`proofpoint_get_forensics`, `recordedfuture_intelligence`, `core_run_script_delete_file`.

#### Foundation normalizers ‚Äî `continueonerror` missing on `Set normalization_source` tasks

`continueonerror: true` was missing from the `Set normalization_source` task in all six
Foundation normalizer playbooks (`Email`, `Endpoint`, `Identity`, `Network`, `Cloud`,
`Generic`), plus `Foundation_-_Enrichment_V3`, `Foundation_-_Product_Classification_V3`,
and `Foundation_Common_-_Extract_Indicators_from_alerts`. A failure on any of these tasks
would halt the Upon Trigger chain. All normalizer tasks now have `continueonerror: true`.

#### SOC_Containment_V3 / SOC_Identity_Containment_V3 ‚Äî Direct `issue.*` reads

Three playbooks were reading `issue.*` fields directly, violating the contract that only
Foundation touches `issue.*`:

- `SOC_Containment_V3`: `issue.username` ‚Üí `SOCFramework.Artifacts.UserName`
- `SOC_Identity_Containment_V3`: `issue.username` ‚Üí `SOCFramework.Artifacts.UserName`,
  `issue.domain` ‚Üí `SOCFramework.Artifacts.Domain`,
  `issue.mitreattcktechnique` ‚Üí `SOCFramework.Mitre.Technique.ID`
- `SOC_Endpoint_Compromise_Evaluation_V3`: three `issue.*` SHA256 input defaults
  ‚Üí `SOCFramework.Artifacts.File`

#### SOC_Email_Exposure_Evaluation_V3 ‚Äî Missing Analysis namespace handoff

`Email.Exposure.recipient_scope` was set by Exposure Evaluation but never copied to
`Analysis.Email.RecipientScope`. `SOC_Email_Spread_Evaluation_V3` reads
`inputs.RecipientScope` which maps to `${Analysis.Email.RecipientScope}`, so every
Spread Evaluation run received an empty input and defaulted to `single_entity` regardless
of actual blast radius.

New task 24 added: `Set Analysis.Email.RecipientScope` ‚Äî runs at the convergence point
after the targeted/broad scope branches, copies `Email.Exposure.recipient_scope` into
the Analysis namespace before the HVU check.

#### SOC_Email_Spread_Evaluation_V3 ‚Äî Null-input pause on numeric condition

Task 2 `Map Recipient Scope to Spread Level` had `tenant_wide AND MailboxCount >= threshold`
as a single AND group. When `MailboxCount` was null, XSIAM could not evaluate the numeric
comparison and surfaced a manual analyst action instead of auto-routing. Two fixes:

1. Condition restructured into two OR groups: `tenant_wide` alone (no numeric check) and
   `multi_user AND MailboxCount >= threshold` (numeric check only evaluated when string
   match passes first).
2. `MailboxCount` input default now uses `SetIfEmpty` to guarantee `"0"` when
   `Email.Exposure.mailbox_count` is absent.

#### SOC_Email_Verdict_Resolution_V3 ‚Äî Null-input pause on click/delivered count conditions

`ClickCount` and `DeliveredCount` inputs now default to `"0"` via `SetIfEmpty` when
`Email.Exposure.click_count` / `Email.Exposure.delivered_count` are absent. Prevents
`greaterThan 0` conditions from pausing for analyst input on alerts where exposure events
have not yet been retrieved.

#### SOC_Email_Signal_Characterization_V3 / SOC_EndPoint_Spread_Evaluation_V3

`continueonerror: true` added to all `SetAndHandleEmpty` tasks that were missing it.
