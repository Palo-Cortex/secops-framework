## SOC Framework NIST IR – Release Notes

### Version 1.3.0

#### Overview

This release delivers the **Email NIST IR lifecycle** to production readiness. It resolves
analyst-blocking pauses in the email analysis chain, closes a context handoff gap between
Exposure Evaluation and Spread Evaluation, fixes `issue.*` contract violations in Containment
playbooks, and hardens all email and endpoint analysis sub-playbooks against null-input
failures.

---

### Bug Fixes

#### SOC_Email_Exposure_Evaluation_V3 — Missing Analysis namespace handoff (Critical)

`Email.Exposure.recipient_scope` was set correctly by Exposure Evaluation but never copied
to `Analysis.Email.RecipientScope`. `SOC_Email_Spread_Evaluation_V3` reads its
`RecipientScope` input from `${Analysis.Email.RecipientScope}`, so every Spread Evaluation
run received an empty value and silently defaulted to `single_entity` regardless of actual
blast radius.

New task 24 added at the convergence point after the targeted/broad scope branch: `Set
Analysis.Email.RecipientScope` copies `Email.Exposure.recipient_scope` into the Analysis
namespace before control passes to the High Value User check. `continueonerror: true` — this
task can never halt the chain.

#### SOC_Email_Spread_Evaluation_V3 — Null-input pause blocking analyst workflow (Critical)

Task 2 `Map Recipient Scope to Spread Level` had `tenant_wide AND MailboxCount >= threshold`
as a single AND condition group. When `MailboxCount` was null — which occurs in every run
where Exposure Evaluation has not yet populated the count — XSIAM could not evaluate the
numeric comparison and surfaced a manual analyst action instead of auto-routing. This caused
the email chain to stop and wait for a human at the Spread Evaluation entry point, violating
the Foundation non-blocking contract.

Two fixes applied:

1. Condition restructured into two OR groups: group 1 is `RecipientScope == tenant_wide`
   alone with no numeric check; group 2 is `RecipientScope == multi_user AND MailboxCount >=
   threshold` where the numeric comparison is only evaluated after the string match passes.
   When `RecipientScope` is empty both string comparisons fail cleanly and `#default#` routes
   to `single_entity` with no analyst prompt.
2. `MailboxCount` input default updated to use `SetIfEmpty` with value `"0"`, guaranteeing
   a valid numeric value is always available.

#### SOC_Email_Verdict_Resolution_V3 — Null-input pause on count conditions

`ClickCount` and `DeliveredCount` inputs now use `SetIfEmpty` with default `"0"` when
`Email.Exposure.click_count` and `Email.Exposure.delivered_count` are absent from context.
Without this, the `greaterThan 0` conditions in task 14 `Recommend Action?` produced the
same analyst-blocking null-input pause as the Spread Evaluation bug above.

#### SOC_Containment_V3 — Direct `issue.*` read (Contract violation)

Task 10 passed `${issue.username}` directly as the `Username` input to
`SOC_Identity_Containment_V3`. Only Foundation playbooks may read `issue.*` fields.

Fix: changed to `${SOCFramework.Artifacts.UserName}`, which Foundation - Normalize
Artifacts writes before the lifecycle runs.

#### SOC_Identity_Containment_V3 — Three direct `issue.*` reads (Contract violation)

Three `issue.*` reads removed:

- Tasks 164 and 166 condition left-hand values: `issue.mitreattcktechnique` →
  `SOCFramework.Mitre.Technique.ID`
- `Username` input default: `${issue.username}` → `${SOCFramework.Artifacts.UserName}`
- `IAMUserDomain` input default: `${issue.domain}` → `${SOCFramework.Artifacts.Domain}`

#### SOC_Endpoint_Compromise_Evaluation_V3 — Direct `issue.*` SHA256 input defaults

Three input defaults sourced raw `issue.*` SHA256 fields directly:
`initiator_sha256`, `xdm_sourceprocess_executable_sha256`, `cgo_sha256`. These were
replaced with `${SOCFramework.Artifacts.File}`, consistent with the file artifact contract
established in 1.1.0.

#### SOC_Email_Signal_Characterization_V3 / SOC_EndPoint_Spread_Evaluation_V3 — Missing `continueonerror`

`continueonerror: true` added to all `SetAndHandleEmpty` tasks that were missing it in both
playbooks. A failure on any set task in an analysis sub-playbook would previously halt the
analysis chain and leave downstream phase playbooks without the context keys they require.

---

### Notes

- The `issue.*` contract violations fixed in this release follow the same pattern audited in
  1.1.0 for Endpoint. A full audit of all remaining NIST IR playbooks is recommended before
  1.4.0 to confirm no further `issue.*` reads exist outside Foundation.
- The Spread Evaluation null-pause fix changes condition evaluation logic. If you have
  custom `RecipientScope` values beyond `tenant_wide`, `multi_user`, and `single_user`,
  verify they route correctly via the `#default#` branch.
