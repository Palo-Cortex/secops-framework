# SocFrameworkCrowdstrikeFalcon — Release Notes v1.1.4

## Summary

Consolidates 15 per-tactic CrowdStrike Falcon correlation rules plus one catch-all into a single unified rule. Adds layout, layout rule, display scripts, and playbook carried from the legacy `soc-crowdstrike-falcon-1.0.14` pack. Introduces suppression where none existed in the old rules. No backwards-compatibility additions required — the old playbook is fully compatible with the new rule. Modeling rule intentionally excluded pending old pack decommission.

---

## Correlation Rule — SOC CrowdStrike Falcon - Endpoint Alerts (rule v2.0.0)

### What changed

**Consolidation.** Replaces 15 per-tactic rules (`Initial Access`, `Execution`, `Persistence`, `Privilege Escalation`, `Defense Evasion`, `Credential Access`, `Discovery`, `Lateral Movement`, `Collection`, `Exfiltration`, `Command and Control`, `Impact`, `Reconnaissance`, `Resource Development`) plus one catch-all rule with a single rule covering all CrowdStrike Falcon EPP detection events.

**Dynamic MITRE tactic routing.** The old rules hardcoded one tactic per rule via a fixed `alert_category`. The new rule uses `user_defined_category: tactic` — the alert category is set dynamically from the `tactic` field in the CrowdStrike event. A detection maps to whichever tactic CrowdStrike reports, not whichever rule happened to fire. Detections that previously fell through to the catch-all rule now carry their correct tactic context.

**Full 12-tactic MITRE definition block.** The old per-tactic rules each carried only their own tactic's `mitre_defs`. The new rule carries a complete block covering all 12 MITRE ATT&CK tactics and their associated techniques, enabling cross-tactic correlation and enrichment.

**Normalized MITRE field names.** The old rules mapped `mitretacticid` to raw `tactic_id` and `mitretacticname` to raw `tactic`. The new rule maps these to the XQL-computed normalized fields `mitre_tactic_id` and `mitre_tactic` (produced by the MITRE normalization alters in the XQL), consistent with the XSIAM data model.

**Alert name format.** Changed from `CrowdStrike Falcon - $alert_name` to `SOC CrowdStrike Falcon - $alert_name`. The layout routes on the ingest tag `DS:CrowdStrike/Falcon_Event` (set at ingest by the CrowdStrike integration), not on alert name — no layout routing impact.

### Suppression

| | Old rules (1.0.14 — all 15) | New rule (v2.0.0) |
|---|---|---|
| Suppression enabled | No — disabled on all 15 rules | Yes |
| Field | — | `composite_id` |
| Duration | — | 1 hour |
| Scope | — | Single detection event only |
| Cross-detection impact | — | None — different detections have different `composite_id` |

`composite_id` is CrowdStrike Falcon's unique identifier per detection event. Two detections on the same host have different `composite_id` values and each fire independently — there is no blast-radius risk. The 1-hour window covers back-to-back polling cycles returning the same detection before it is acknowledged. This is a net improvement: the 1.0.14 rules had no deduplication at all, allowing duplicate alerts from repeated polling of the same detection.

### Incident fields — none required

The 1.0.14 pack contains no incident field definitions. CrowdStrike-specific incident fields are owned by the CrowdStrike Falcon integration content pack installed from the XSIAM Marketplace.

### Modeling rule — intentionally excluded

The 1.0.14 pack contains one modeling rule on `crowdstrike_falcon_event_raw`. XSIAM does not document behavior when two separately installed packs both register a User Defined modeling rule on the same dataset. Since `soc-crowdstrike-falcon-1.0.14` must remain installed during the migration window, its modeling rule provides XDM coverage. Adding a second rule creates an undefined collision. Modeling rule will be added at v1.2.0 once the old pack is decommissioned.

## Decommission Path

At v1.2.0, when the old pack is ready to retire:

1. V3 endpoint chain confirmed active on all tenants replacing `CrowdStrike_Basic_Alert_Handler`
2. Add `SOCCrowdStrikeFalconModelingRules` from old pack to this pack
3. Bump to v1.2.0
4. Uninstall `soc-crowdstrike-falcon-1.0.14`

## Breaking Changes

None. All 15 old rules remain installed and can coexist with the new rule while `is_enabled: false` is set on the new rule. Cut over by flipping `is_enabled: true` and disabling all 15 old rules plus the catch-all. The old rules use the `CrowdStrike Falcon - ` alert name prefix; the new rule uses `SOC CrowdStrike Falcon - `. If any alerting rules, dashboards, or saved searches filter on the alert name prefix, update those filters at cutover.
